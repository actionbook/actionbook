# Design: Session + Daemon Management Architecture for CLI v1.0.0

## Problem Statement

actionbook-rs v0.x 的 session + daemon 管理存在 3 个根因性架构缺陷（详见 root-cause-analysis.zh.md）：
1. 错误的执行上下文抽象 - 5 种 browser mode 混合在一个 BrowserMode enum 里
2. 错误的权威抽象 - 无单一 authority，文件系统作为 IPC，ambient trust
3. 错误的语义抽象 - 无共享领域模型，字符串代替类型，无显式状态机

v1.0.0 需要从根因出发重新设计 session + daemon 的管理架构。

## What Makes This Cool

这个设计一旦落地，actionbook 将是唯一一个用统一架构同时支持本地浏览器、用户浏览器（extension）、云浏览器（remote WSS）的 AI browser automation 工具。CLI 变成约 500 行的薄客户端，daemon 成为可服务 CLI/MCP/AI SDK 多端的 browser control plane。

## Constraints

- 来自 PRD：无状态 CLI，有状态 daemon，绝对路径寻址（--session + --tab），短 ID（s0, t3）
- 来自 founder：
  1. 所有连接都走 daemon，无 bypass
  2. local/extension/cloud 统一架构，差异仅在 backend adapter
  3. session + tab 是最基础原语；profile 仅是配置
  4. CLI 尽量无状态
- 来自 core_design.md：类型化协议（v2 Action enum），错误即引导（hint 字段），干净断裂

## Premises

1. Daemon 是唯一运行时权威 - 所有模式都走 daemon，无 --no-daemon fallback
2. Session 是运行时原语，Profile 仅是配置 - session ID 是 daemon 生成的短 ID（s0, s1），profile 只决定启动参数
3. 一个全局 daemon 管理所有 session - 重启后从持久化文件恢复
4. Thick Daemon + 类型化 Action 协议 - CLI 是纯无状态 RPC 客户端，daemon 持有所有 browser 逻辑，无 raw CDP 透传

## Cross-Model Perspective

Codex 独立审视此设计后提出 5 个洞察：
- 最强版本：推进到 capability-based, event-sourced control plane，Session actor 是权威状态机
- 致命盲点：寻址做对了但授权没做完，--session s0 --tab t3 不是权限模型，需要 exclusive lease
- 协议：长度前缀分帧 + CBOR 优于 line-delimited JSON；需要背压、deadline、取消语义
- 持久化：control-plane state 持久化，runtime garbage 不持久化；crash 后 session 标 Recovering
- Backend trait：不要把 40 个 Action 塞进 trait，拆成 Factory + Session handle，daemon 编译 Action 到 BackendOp

## Recommended Approach: Per-Session Async Task + Thick Daemon

### 核心设计原则

P1: Daemon 是唯一权威（Single Authority）
    所有浏览器操作必须通过 daemon。CLI、MCP、AI SDK 都是 daemon 的客户端。
    不存在绕过 daemon 直连浏览器的路径。

P2: 无状态接口，有状态运行时（Stateless Interface, Stateful Runtime）
    CLI 是纯无状态的 RPC 客户端。所有运行时状态（session 注册表、
    CDP 连接、tab 路由）由 daemon 独占管理。

P3: 统一架构，差异在 adapter（Unified Core, Adapter Differences）
    local/extension/cloud 共享一个 session 管理架构。
    差异封装在 BrowserBackend trait 的实现中，对 CLI 和 daemon 核心逻辑透明。

P4: 绝对路径寻址（Absolute-Path Addressing）
    每条 per-tab 命令必须带 --session + --tab，无隐式"当前 tab"。
    消除全局锁，多 tab 并行操作成为一等公民。

P5: Thick Daemon - CLI 是薄客户端（Thin CLI, Thick Daemon）
    所有 browser 逻辑（CDP 命令构造、backend 路由、stealth 注入）在 daemon 内。
    CLI 只做 arg parse -> Action 序列化 -> 发送 -> 打印响应。约 500 行。
    daemon 是可服务多端（CLI/MCP/AI SDK）的 browser control plane。

P6: 类型化协议，无 raw CDP（Typed Protocol, No Raw CDP）
    CLI-Daemon 协议是类型化的 Action enum（约 40 个变体）。
    不提供 raw CDP 透传。错误按恢复策略分类（Retryable/UserAction/Fatal）。

P7: Session 内串行，Session 间并行（Serial Within, Parallel Across）
    每个 session 是一个 tokio task，持有自己的 backend 连接。
    session 内命令通过 channel 序列化，无竞态。
    session 间完全独立，互不阻塞。

**关键点：三个 Session Actor 的内部结构完全一致**（Action Handler + Tab/Window Registry），
唯一的区别是 `BackendSession` 的实现（`impl`）不同：

| | Local | Extension | Cloud |
|---|---|---|---|
| BackendSession impl | `LocalBackend` | `ExtensionBackend` | `CloudBackend` |
| 连接方式 | 直接 WS 到 Chrome | 通过 daemon 内嵌的 ExtensionBridge | 直接 WSS 到云端 |
| exec(BackendOp) | WS 发 CDP JSON | WS Bridge -> Extension -> chrome.debugger | WSS 发 CDP JSON |
| Chrome 进程 | daemon 启动 | 用户已有 | 云端管理 |

ExtensionBackend 详细说明：

```
ExtensionBackend.exec(op: BackendOp) 的内部实现：

  1. 将 BackendOp 序列化为 CDP JSON
  2. 通过 ExtensionBridge 的 WS 连接发送给 Chrome Extension
  3. Extension 收到后调用 chrome.debugger.sendCommand(targetId, method, params)
  4. Chrome 执行并返回结果
  5. Extension 将结果通过 WS 返回给 ExtensionBackend
  6. ExtensionBackend 将结果解析为 OpResult 返回

  对 Session Actor 来说，调用 backend.exec(op) 和 Local/Cloud 模式完全一样，
  它不知道底层多了一层 Extension 中转。
```

```rust
// 三种 BackendSession 实现的 exec 方法对比（伪代码）：

impl BackendSession for LocalBackend {
    async fn exec(&mut self, op: BackendOp) -> Result<OpResult> {
        // 直接通过 WS 发 CDP 命令到本地 Chrome
        let cdp_msg = op.to_cdp_json();
        self.ws.send(cdp_msg).await?;
        let resp = self.ws.recv().await?;
        OpResult::from_cdp(resp)
    }
}

impl BackendSession for ExtensionBackend {
    async fn exec(&mut self, op: BackendOp) -> Result<OpResult> {
        // 通过 ExtensionBridge WS 发给 Chrome Extension
        // Extension 再通过 chrome.debugger API 执行
        let bridge_msg = BridgeMessage::Exec { target: self.target_id, op };
        self.bridge_ws.send(bridge_msg).await?;
        let resp = self.bridge_ws.recv().await?;
        OpResult::from_bridge(resp)
    }
}

impl BackendSession for CloudBackend {
    async fn exec(&mut self, op: BackendOp) -> Result<OpResult> {
        // 通过 WSS 发 CDP 命令到远端浏览器
        let cdp_msg = op.to_cdp_json();
        self.wss.send(cdp_msg).await?;
        let resp = self.wss.recv().await?;
        OpResult::from_cdp(resp)
    }
}
```

### 领域模型

```rust
/// Profile — 配置层，决定如何启动/连接浏览器
struct Profile {
    name: String,
    mode: Mode,                         // Local | Extension | Cloud
    executable_path: Option<String>,    // Local only
    headless: bool,                     // Local only
    user_data_dir: Option<String>,      // Local only
    cdp_endpoint: Option<String>,       // Cloud only (wss://...)
}

/// Session — 运行时层，daemon 管理的核心原语
struct SessionEntry {
    id: SessionId,                      // s0, s1, s2... (daemon 生成)
    uuid: Uuid,                         // 持久化用，crash recovery 关联
    profile: String,                    // 引用 Profile 配置
    backend: Box<dyn BackendSession>,   // 实际的浏览器连接
    state: SessionState,                // 状态机：Starting/Ready/Executing/...
    windows: HashMap<WindowId, WindowEntry>,
    tabs: HashMap<TabId, TabEntry>,
    created_at: Instant,
}

/// Window — tab 的分组
struct WindowEntry {
    id: WindowId,                       // w0, w1, w2...
    chrome_window_id: i64,              // CDP 返回的 window ID
    tabs: Vec<TabId>,                   // 该窗口下的 tab
}

/// Tab — session 内的标签页
struct TabEntry {
    id: TabId,                          // t0, t1, t2...
    target_id: String,                  // CDP target ID
    window: WindowId,                   // 所属窗口
    url: String,
    title: String,
}

/// Mode — 三种浏览器连接模式
enum Mode {
    Local,      // 启动本地 Chrome
    Extension,  // 连接用户的 Chrome（via extension bridge）
    Cloud,      // 连接远端 WSS endpoint
}
```

Profile 语义差异：

| Mode | Profile : Session | 原因 |
|------|-------------------|------|
| Local | 1 : 1 | Chrome 限制同名 user-data-dir 不能被两个进程使用 |
| Extension | 1 : 1 | 用户只有一个浏览器实例 |
| Cloud | 1 : N | 不同 WSS 实例是独立的浏览器，浏览器层已隔离 |

#### ActionResult（按恢复策略分类）

```rust
enum ActionResult {
    /// 成功，data 是命令特定的结构化返回
    Ok { data: Value },
    /// 暂时失败，可重试（如网络抖动、CDP 超时）
    Retryable { reason: String, hint: String },
    /// 需要用户介入（如 extension 未连接、浏览器被关闭）
    UserAction { action: String, hint: String },
    /// 不可恢复（如 session 不存在、参数非法）
    Fatal { code: String, message: String, hint: String },
}
```

所有错误变体都包含 `hint` 字段，告诉 agent 下一步该做什么。
`code` 是机器可读的（如 `"session_not_found"`），`message` 是人类可读的。

Layer 3: Wire Protocol（传输层）

Transport: UDS (Unix) / Named Pipe (Windows)
Framing:   长度前缀 (4 bytes LE length + payload)
Encoding:  serde + JSON (v1.0，便于调试和多语言 SDK)

Request:  { "v": 2, "id": 42, "action": { ... } }
Response: { "id": 42, "result": { ... } }

### BrowserBackend trait 设计

```rust
/// Factory — 创建/附加/恢复 session
trait BrowserBackendFactory: Send + Sync {
    fn kind(&self) -> BackendKind;          // Local | Extension | Cloud
    fn capabilities(&self) -> Capabilities; // 哪些 BackendOp 受支持

    async fn start(&self, spec: StartSpec) -> Result<Box<dyn BackendSession>>;
    async fn attach(&self, spec: AttachSpec) -> Result<Box<dyn BackendSession>>;
    async fn resume(&self, cp: Checkpoint) -> Result<Box<dyn BackendSession>>;
}

/// Live session handle — daemon 持有，per-session task 独占
trait BackendSession: Send {
    /// 异步事件流：disconnect, target created/destroyed, dialog, download
    fn events(&mut self) -> BoxStream<'static, BackendEvent>;

    /// 执行 CDP-level 操作（BackendOp）
    async fn exec(&mut self, op: BackendOp) -> Result<OpResult>;

    /// 列出所有 CDP targets（tabs + workers）
    async fn list_targets(&self) -> Result<Vec<TargetInfo>>;

    /// 生成可持久化的恢复快照
    async fn checkpoint(&self) -> Result<Checkpoint>;

    /// 健康检查
    async fn health(&self) -> Result<Health>;

    /// 关闭 session
    async fn shutdown(&mut self, policy: ShutdownPolicy) -> Result<()>;
}
```

### Persistence and Crash Recovery

Durable state (写入 ~/.actionbook/daemon-state.json):
  session UUID + short alias, backend kind, profile name,
  tab alias + stable target key,
  backend-specific checkpoint:
    Local:     browser PID, WS URL, user-data-dir
    Extension: bridge port, extension ID
    Cloud:     WSS endpoint, auth headers, resume token

Ephemeral state (不持久化):
  WebSocket connection, CDP session ID, DOM node/object handles,
  Frame tree cache, Pending command queue, Snapshot ref cache

Crash recovery:
  daemon 重启 -> 读 daemon-state.json
  -> 每个 session 标为 Recovering
  -> 按 backend 类型 reconcile:
    Local recovery decision tree:
      1. 检查 PID 是否存活 AND 是否是 Chrome 进程
      2. 尝试旧 WS endpoint 重连
      3. WS 失败但 PID 是 Chrome -> 通过 /json/version 重新发现 WS URL
      4. 都失败 -> 标为 Lost
    Extension: 等 bridge heartbeat (30s) -> 成功=Ready, 超时=UserAction
    Cloud:     用 resume token 重连 -> 成功=Ready, 不支持 resume -> Lost
  -> Lost session 保留 alias 但标记不可用
  -> 不自动重放未完成的 mutating command

Session ID 说明:
  SessionId 是 daemon 实例内单调递增的短别名（s0, s1, s2...）
  内部由 UUID 支撑（用于 crash recovery 后关联持久状态）
  wire protocol 中使用短别名（s0）寻址

### Session-Profile 约束

Local:     1 profile name = max 1 active session (Chrome 限制)
Extension: 1 extension backend = max 1 active session
Cloud:     1 profile = N sessions（不同 endpoint，浏览器层已隔离）

### Daemon Auto-Start Protocol

CLI 发现 daemon 未运行时的处理流程：
1. CLI 尝试连接 UDS (~/.actionbook/daemon.sock)
2. 连接失败 -> CLI fork 启动 daemon 进程（等待 daemon ready signal）
3. Daemon 启动后写 PID + 绑定 UDS + 读取持久化状态 -> signal ready
4. CLI 重试连接 -> 成功后发送 Action
5. 版本不匹配（CLI v2 protocol, daemon v1）-> daemon 自行重启升级

Extension 模式架构（WS + Token 混合方案）：

Extension 连接分两个阶段：先通过 Chrome native messaging 安全通道获取一次性 token，
再用 token 建立持久 WS 连接。

```
Phase 1: 一次性握手（Chrome Native Messaging 安全通道）

  Chrome Extension             Native Messaging Host            Daemon
       |                       (actionbook binary)                |
       |                              |                           |
       |-- connectNative() ---------> |                           |
       |                              |-- UDS connect ----------->|
       |                              |                           |
       |                              |-- "get_bridge_info" ----->|
       |                              |                           |
       |                              |<-- { port: 19222,    ----|
       |                              |      token: "a8f3..." }  |
       |<-- { port, token } --------- |                           |
       |                              |                           |
       |   (host 退出，native messaging 连接关闭)                  |


Phase 2: 持久 WS 连接（token 认证）

  Chrome Extension                          Daemon (ExtensionBridge)
       |                                         |
       |-- ws://localhost:19222 ---------------->|
       |   Header: Authorization: Bearer a8f3..  |
       |                                         |
       |                                         |-- 验证 token
       |                                         |   (一次性，用后作废)
       |                                         |
       |<-- { type: "connected", session: "s0" } |
       |                                         |
       |   (持久双向 WS，后续所有通信走此连接)       |
       |                                         |
       |<-- BackendOp::GetDocument ------------- |  (daemon 发命令)
       |                                         |
       |-- chrome.debugger.sendCommand() --> Chrome
       |   <---- result ----                     |
       |                                         |
       |--- { result: ... } ------------------->|  (extension 回结果)
```

```
                                  Daemon 进程
                     +-------------------------------------+
                     |                                     |
 CLI --UDS-->        |  Request Router                     |
                     |       |                             |
                     |       v                             |
                     |  Session Actor (s0)                 |
                     |       |                             |
                     |  Action Handler                     |

                     |       | localhost:PORT              |
                     +-------|---------+-------------------+
                             |         ^
                   WS 连接   |         | Native Messaging
                  (一次性     |         | (Phase 1: 获取
                   token     v         |  port + token)
                   认证)     |         |
                     +-------|---------+-------------------+

                     |       |                             |

                     |       |                             |
                     +-------|---------+-------------------+
                             |

```

安全模型：

```
为什么比 v0.x 安全：

  v0.x:  Extension 连 bridge → 验证 Origin header → 可伪造
  v1.0:  Extension 连 daemon → 需 token → token 仅通过 native messaging 发放

  Native messaging 是 Chrome 的受信通道：
  1. Chrome 只启动注册在 NativeMessagingHosts/ 的 host binary
  2. 只有 allowed_origins 中的 extension 才能 connectNative()
  3. host 连接 daemon UDS 获取一次性 token
  4. token 通过 native messaging 安全传回 extension
  5. extension 用 token 建立 WS 连接到 daemon
  6. daemon 验证 token（一次性，用后作废，无法重放）
  7. 其他本地进程无法获取 token → 无法冒充 extension

  安装需求（actionbook setup 已有此功能）：
  - NativeMessagingHosts manifest 注册
  - host binary = actionbook 本身（复用 native-messaging 子命令）
```

- v1.0 限制：1 extension = 1 session（产品选择，非架构限制）

### Concurrency Model (v1.0)

v1.0: 同一 session 的命令按 FIFO 队列串行执行。
多个 CLI 客户端可以同时发送命令到同一 session，命令按到达顺序排队。
不实现 exclusive lease（v1.5 再做）。
不自动拒绝并发客户端。

### iFrame Handling (v1.0)

v1.0 策略：默认展开 iframe 内容到 tab 的 snapshot 和 DOM 操作中。
即 iframe 内容是 tab 的一部分，不需要 frame ID 寻址。
如果 CDP 层无法完全支持（如 cross-origin iframe 限制），降级为标注 [iframe: url] 占位。
后续版本可考虑 --frame 参数做精确寻址。

### Window 管理

Window 是 tab 的分组概念。一个 session 有多个 window，一个 window 有多个 tab。

```
Session (s0)
├── Window (w0, 默认窗口)
│   ├── Tab t0 (about:blank)
│   └── Tab t1 (https://example.com)
└── Window (w1, 用户通过 --new-window 创建)
    └── Tab t2 (https://github.com)
```

Window Actions：

```rust
enum Action {
    // ... 其他 Action ...

    // Window management (require --session)
    ListWindows { session: SessionId },
    // new-tab 和 open 支持 --new-window 和 --window 参数
    NewTab { session: SessionId, url: String, new_window: bool, window: Option<WindowId> },
    Open { session: SessionId, url: String, new_window: bool, window: Option<WindowId> },
}
```

领域模型中的 Window：

```rust
struct WindowEntry {
    id: WindowId,         // w0, w1, w2... daemon 生成
    chrome_window_id: i64, // CDP Browser.getWindowForTarget 返回的 ID
    tabs: Vec<TabId>,     // 该窗口下的 tab 列表
}
```

Window ID（w0, w1）是 daemon 内生成的短别名。
Tab 在创建时归属到某个 window。`--new-window` 创建新 window 并在其中打开 tab。
默认不指定 window 时，tab 创建在 session 的默认窗口（w0）中。
